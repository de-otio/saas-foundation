# Design spike: app-signed session cookie for edge auth

**Status:** spike (evaluation only — no code changes proposed here)
**Scope:** `@de-otio/vestibulum` / `@de-otio/vestibulum-cdk`, the
`MagicLinkAuthSite` construct and its `check-auth` Lambda@Edge verifier.
**Relationship to other work:** the deploy-time custom-resource string-injection
fix (tracked separately) is the *tactical* fix for the same root cause. This
spike evaluates the *strategic* alternative. The two are mutually exclusive at
steady state but the CR fix is a viable stepping-stone — see §6.

---

## 1. Problem recap

`check-auth` runs on the CloudFront **viewer-request** of every protected
request. Today it verifies the **raw Cognito ID token** (cookie `id-token`)
with `aws-jwt-verify`, configured against the pool's JWKS:

```
CognitoJwtVerifier.create({ userPoolId, tokenUse: "id", clientId, ... })
```

That configuration needs two concrete values baked into the edge bundle:
`<pool-id>` and the website `<client-id>`. Both are only known at
**deploy time** — the consumer (`atrium`) wires identity across stacks via SSM,
so in CDK they are CloudFormation **tokens**, not literals.

Lambda@Edge makes this hard in two compounding ways:

| Constraint | Consequence |
|---|---|
| **No environment variables** on Lambda@Edge | Config can't be passed at runtime the way `auth-verify` gets it. |
| **Static asset only** — the bundle is replicated to edge regions ahead of invocation | A CFN token can't survive into the asset; it must be a concrete string at synth/deploy. |

The current source carries a `// __VESTIBULUM_CONFIG_START__ … END__` placeholder
block that a deploy-time step **string-injects** with concrete values. It works,
but it means:

1. The edge bundle's content is **mutated per deployment** (the asset hash is no
   longer a pure function of source — it depends on injected identity).
2. The edge holds a **network dependency on Cognito's JWKS** endpoint
   (`https://cognito-idp.<region>.amazonaws.com/<pool-id>/.well-known/jwks.json`),
   refreshed on a 15-min TTL, fetched from whichever edge region served the
   request. That is a cross-region, third-party (from the edge's POV) call on a
   hot path, and JWKS-fetch failure is a fail-closed `302 /login`.
3. The verifier is **coupled to Cognito's token format** (RS256, pool issuer,
   `aud == clientId`), so any change to the identity provider ripples to the edge.

Root cause: **the edge is being asked to validate a token minted by a system
whose config it doesn't natively know.** Both the CR fix and this spike attack
that; they differ in *what the edge validates*.

---

## 2. Proposed design: app-signed session token

Move the trust boundary. The regional `auth-verify` Lambda already holds env
config and Cognito access; let it **exchange** the verified Cognito ID token for
a short-lived **app-signed session token** (call it the *AST*), set as the
session cookie. The edge then verifies the **AST**, not the Cognito token.

```
            Cognito RespondToAuthChallenge
  browser ───────────────▶ auth-verify (regional)
                              │  verifies Cognito IdToken (already does)
                              │  mints AST  = sign(claims, app_private_key)
                              ▼
  browser ◀──── Set-Cookie: app-session=<AST>  (HttpOnly, Secure, SameSite=Lax)
                              │
  browser ───────────────▶ check-auth @ edge
                              verify(AST, app_PUBLIC_key)   ← baked, concrete
```

The **public key** is the only thing baked at the edge. It is a concrete,
deploy-known value (generated by/for the deploy), **not a Cognito token**, so the
pool/client-config baking problem disappears. The edge never talks to Cognito.

### 2.1 Token format

Prefer a **compact JWS** with an **asymmetric** signature so the edge holds only
a public key:

| Choice | Recommendation | Rationale |
|---|---|---|
| Algorithm | **EdDSA (Ed25519)**, fallback **ES256** | Tiny keys/signatures, fast verify, constant-time libraries widely available. Avoids RSA's large keys. ES256 if the chosen verify library lacks solid EdDSA support in the edge runtime. |
| Encoding | Compact JWS (`header.payload.signature`) | Same shape `aws-jwt-verify`-style verifiers already parse; cookie-sized. |
| **Not** | HS256 / any symmetric | Symmetric means the *verify* key == *sign* key; baking it at the edge re-creates a secret-distribution problem and a key-confusion risk. The whole point is asymmetric. |
| **Not** | Opaque/stateful session id | Would require an edge→datastore lookup — worse than the JWKS dependency we're removing. |

Pin `alg` at the edge to the single expected value (mirroring the existing
RS256-only allow-list that guards `alg:none`/HS-confusion). No `jku`/`x5u`
header processing — the verify key is baked, never fetched.

### 2.2 Claims

Carry exactly what downstream consumers need and nothing more:

| Claim | Source | Purpose |
|---|---|---|
| `sub` | Cognito `sub` | Stable user id. |
| `iss` | fixed app issuer string, e.g. `vestibulum:<env>` | Pinned at edge. |
| `aud` | the site/distribution id | Pinned at edge; scopes the token to this app. |
| `exp` | now + **5–15 min** | Short TTL = the revocation story (§4). Match the current `ID_TOKEN_MAX_AGE` (15 min) or go tighter. |
| `iat`, `nbf` | now | Clock-skew window. |
| `custom:tenant_id` → `tid` | currently on Cognito ID token (pre-token-generation injects it) | Load-bearing for the multi-tenant edge **structural Host↔tenant binding** (`shared-distribution/edge/check-auth.ts`). Must be re-minted into the AST verbatim. |
| `read_spaces` (and any per-space claim) | currently on Cognito ID token | Consumed by the **per-space origin-request gate** that trust-decodes the cookie. Must survive the exchange (§5). |

`auth-verify` copies these from the verified Cognito claims into the AST it
signs. The AST is **app-controlled**, so the claim namespace can be normalized
(drop the `custom:` prefix) — but that's a downstream-contract change; default to
**preserving names** to minimize migration churn.

### 2.3 Where signing happens

`auth-verify` (and any sibling that sets the session cookie — `auth-login` if it
also lands an authenticated session). It already:

- verifies the Cognito result (`RespondToAuthChallenge`),
- has env config + IAM,
- runs regionally (can hold a private key / call KMS).

It gains: load the signing key (or a KMS key id) from env/secret, sign, set
`app-session` cookie instead of (or alongside, during migration) `id-token`.

### 2.4 Cookie attributes

Keep the current hardened posture from `auth-verify/cookie.ts`:

```
Set-Cookie: app-session=<AST>; HttpOnly; Secure; SameSite=Lax; Path=/;
            Domain=<domain>; Max-Age=<exp seconds>
```

`SameSite=Lax` (matches today's `id-token`); `HttpOnly` (no JS access);
`Max-Age` == AST `exp`. The Cognito **refresh-token** cookie keeps its current
`SameSite=Strict; Path=/auth-verify` scope — refresh still mints a fresh AST.

## 3. Key management

This is the crux of the trade-off: we **add a signing key to protect**. Three
options for where the private key lives:

| Option | Private key location | Edge gets public key via | Pros | Cons |
|---|---|---|---|---|
| **A. KMS asymmetric** (`SIGN_VERIFY`, Ed25519/ECC_NIST_P256) | KMS, never leaves | CDK reads `GetPublicKey` at deploy → bakes PEM into edge bundle | Key never in app memory; rotation = new KMS key; audited via CloudTrail | `auth-verify` makes a **KMS `Sign` call per login** (latency + cost on the login path, not the hot path — acceptable); deploy needs `GetPublicKey` |
| **B. App-held keypair** (private key in Secrets Manager) | Secrets Manager, loaded into `auth-verify` memory | CDK reads the public half from the secret at deploy → bakes it | No per-sign network call; fastest signing | Private key in Lambda memory = larger exposure; rotation = secret rotation + redeploy; must generate the keypair somewhere trusted |
| **C. CDK-generated keypair at synth** | generated during synth, written to a secret | baked directly from synth output | Simplest wiring | Private key transits the synth machine / CFN; weakest custody; **not recommended** for prod |

**Recommendation: Option A (KMS asymmetric).** Signing is on the *login* path,
not the per-request hot path, so a KMS `Sign` round-trip per login is affordable
(logins are rate-limited already — reserved concurrency 20). The private key
never enters Lambda memory, and rotation is a first-class KMS operation with an
audit trail. Option B is the fallback if KMS `Sign` latency proves material.

### 3.1 Rotation

Because the **public** key is *baked at the edge*, rotation must not assume an
atomic swap — replicated edge functions update on CloudFront's schedule, and old
viewers hold ASTs signed by the old key for up to one `exp` window.

Strategy: **multi-key verify with `kid`.**

1. AST header carries a `kid`. The edge bakes a **set** of accepted public keys
   keyed by `kid` (current + previous), not a single key.
2. Rotation:
   - **(t0)** Generate new key `kid=v2`. Bake **both** `v1` and `v2` public keys
     at the edge; deploy the edge. `auth-verify` still signs with `v1`.
   - **(t1, after edge propagation)** Flip `auth-verify` to sign with `v2`.
   - **(t2, after one `exp` window + margin)** Bake **only** `v2` at the edge;
     deploy. `v1` is now fully retired.
3. The window where both keys verify is bounded by AST `exp` (minutes), not by
   refresh-token lifetime.

This is a `kid`-keyed map baked at the edge — the same shape the deploy-time
injection already produces, just with N keys instead of pool/client strings.

### 3.2 Blast radius

| Event | Blast radius |
|---|---|
| Private key compromise | Attacker can mint valid AST for any tenant until the key is rotated out (bounded by rotation time + one `exp`). **Worse than today in one sense** — today there is no app signing key to steal; the equivalent compromise is the Cognito pool itself, which AWS custodies. This is the honest cost of the design (§4). |
| Public key (edge) leak | No impact — public by definition. |
| KMS unavailable | `auth-verify` `Sign` fails → login fails closed (no AST issued). Hot path unaffected (edge verifies locally). Strictly better than today's edge-side JWKS dependency, which fails the *hot path*. |

## 4. Security analysis vs. current approach

| Dimension | Today (Cognito ID token at edge) | Proposed (AST at edge) | Verdict |
|---|---|---|---|
| **Edge attack surface** | Edge fetches JWKS over the network (cross-region, 15-min TTL); parses RS256 from a remote key set | Edge verifies against a **baked** public key set; **no network** | **Better** — removes a hot-path external dependency and its failure mode |
| **Config exposure** | Pool/client baked via string injection (mutates bundle per deploy) | Only a **public** key baked (safe to expose) | **Better** — public key is non-secret |
| **Algorithm pinning / downgrade** | RS256-only allow-list; `alg:none`/HS-confusion blocked | Single `alg` (EdDSA/ES256) pinned; verify-key is asymmetric so HS-confusion is structurally impossible (no shared secret to confuse) | **Equal/better** |
| **Token theft / replay** | Stolen `id-token` cookie replayable until `exp` (15 min) | Stolen AST replayable until `exp` (5–15 min) | **Equal** — same cookie-theft model (`HttpOnly`/`Secure`/`SameSite` unchanged). Shorter `exp` is the lever; both can use it |
| **Clock skew** | `graceSeconds: 60` at edge | Same `graceSeconds` window on AST `exp`/`nbf` | **Equal** |
| **Revocation** | No real-time revocation; rely on short `exp` + refresh denial | Same — short `exp`; refresh path (in `auth-verify`) is where denylist/global-signout is enforced before re-minting an AST | **Equal** — neither does instant edge revocation; both lean on short `exp` |
| **Keys to protect** | None app-side (Cognito custodies its signing key) | **One new private key** (`auth-verify`/KMS) | **Worse** — net-new secret. Mitigated by KMS custody (Option A) and bounded rotation |
| **Cross-tenant binding** (multi-tenant dist) | Edge enforces `Host ↔ custom:tenant_id` on the Cognito claim | Edge enforces `Host ↔ tid` on the AST claim — **identical logic**, different token | **Equal**, provided `auth-verify` re-mints `tenant_id` **verbatim from the verified Cognito claim** (never from request input) |

**Honest summary:** the proposal is **more secure on the hot path** (no edge
JWKS dependency, public-only baking, structural HS-confusion immunity) but
introduces **one new private key to protect**. It is *not* unconditionally more
secure — it trades a managed-provider signing key (Cognito's, AWS-custodied) for
an app-custodied one. With KMS custody and short `exp`, the trade is favorable
for *this* problem, because the thing we remove (a per-request external network
trust dependency at the edge) is a larger and more failure-prone surface than the
thing we add (a well-custodied, rotatable signing key exercised on a
rate-limited login path).

## 5. Migration path

The cookie consumers are: (a) `check-auth` (single-tenant `MagicLinkAuthSite`),
(b) the **shared-distribution** `check-auth` (multi-tenant, structural binding),
and (c) the **per-space origin-request gate**, which **trust-decodes** the cookie
(reads `read_spaces` *without* re-verifying — it relies on the viewer-request
`check-auth` having already verified). Any migration must keep (c) reading a
decodable token with the same claim names.

### 5.1 Dual-accept window

1. **Phase 0 (today):** edge verifies Cognito `id-token`.
2. **Phase 1 — emit both:** `auth-verify` sets **both** `id-token` (Cognito) and
   `app-session` (AST) cookies. Edge `check-auth` updated to: try `app-session`
   first (verify with baked public key); **fall back** to `id-token`
   (Cognito-verify) if `app-session` is absent/invalid. The origin-request gate
   keeps trust-decoding whichever cookie is present (claim names preserved → no
   gate change needed yet).
3. **Phase 2 — flip primary:** stop verifying `id-token` at the edge; `app-session`
   is required. Keep setting `id-token` only if some other consumer still needs it
   (it shouldn't — the AST carries the same claims). Point the origin-request gate
   at `app-session`.
4. **Phase 3 — retire:** stop setting `id-token`; remove Cognito-verify code from
   the edge; remove the pool/client string-injection. Edge bundle now contains
   only the baked public-key set.

Each phase is independently deployable and reversible. The dual-accept window
must be at least one **refresh-token lifetime** (currently 24 h) so no live
session is stranded holding only the retired cookie.

### 5.2 Interaction with the trust-decoding origin-request gate

The per-space gate decodes-without-verifying **by design** (it trusts the
viewer-request `check-auth` upstream). Two requirements:

- **Claim names must match.** Keep `read_spaces` and `tenant_id` (not
  `custom:tenant_id`→`tid`) on the AST during migration so the gate needs no
  change. Rename later as a separate, gated step if desired.
- **The gate must never become the only verifier.** Its security depends on
  `check-auth` running first on the same request. The behaviour split (edge
  verifies, gate trust-decodes) is preserved — the AST is just a different signed
  envelope carrying the same claims.

### 5.3 Multi-tenant structural binding

`shared-distribution/edge/check-auth.ts` does `claims['custom:tenant_id'] ===
tenantSubdomain`. Migration there is a one-line claim-source change once the AST
carries `tenant_id`; the **load-bearing strict-equality cross-tenant gate is
unchanged**. Its property tests (cross-tenant rejection, wrong-key, expired,
no-token) port directly to the AST verifier.

## 6. Cost / complexity vs. the deploy-time-CR fix, and recommendation

| Axis | Deploy-time CR fix (string-inject pool/client) | App-signed session (this spike) |
|---|---|---|
| **Edge network dependency** | **Keeps** edge→Cognito JWKS (hot-path, fail-closed) | **Removes** it |
| **Edge bundle determinism** | Bundle mutated per deploy with identity strings | Bundle mutated per deploy with **public key(s)** — same mechanism, non-secret payload |
| **New secrets to manage** | None | **One** private signing key (KMS preferred) |
| **New moving parts** | A custom resource + injection step (already in flight) | Signing in `auth-verify`, key gen + rotation pipeline, multi-key edge verify, dual-cookie migration |
| **`auth-verify` change** | None | Adds a sign step (KMS `Sign` or in-memory) per login |
| **Migration risk** | Low — pure config plumbing, no token-format change | Moderate — staged dual-accept rollout touching 3 cookie consumers |
| **Coupling to Cognito** | Edge stays coupled to Cognito token format | Edge decoupled; identity provider swappable behind `auth-verify` |
| **Effort** | **Small** (the CR is mostly done) | **Medium** (keypair + rotation + staged migration) |

### Recommendation: **conditional go — adopt as the strategic target, ship the CR fix first.**

- **Now:** land the **deploy-time CR fix** to unblock deploys. It's small,
  already in flight, and removes the immediate pain (CFN-token-at-edge). It does
  **not** remove the edge→JWKS dependency, but it's the right tactical move.
- **Next (strategic):** pursue the **app-signed session** as the long-term shape.
  Its decisive wins are removing the **hot-path edge→JWKS network dependency** and
  **decoupling the edge from the Cognito token format** — both reduce the per-
  request failure surface and future identity-provider churn. The cost is one
  well-custodied signing key (KMS) and a staged migration, both bounded and
  reversible.
- **Go/no-go gate for the strategic phase:** proceed **only if** (a) KMS `Sign`
  latency on the login path is acceptable under load test (or Option B is chosen),
  and (b) the `kid`-keyed multi-key edge verify + bounded rotation runbook is
  specified and tested before first rotation. If neither the JWKS hot-path
  dependency nor Cognito-format coupling is causing real pain after the CR fix,
  this is a reasonable **defer** — the CR fix is sufficient on its own for a
  single-provider, single-pool deployment.

**One-line verdict:** the CR fix solves the *baking* problem; the app-signed
session additionally solves the *edge-network-dependency* and *provider-coupling*
problems at the cost of one new key. Ship the CR fix now; adopt app-signed
sessions when edge-side Cognito coupling or JWKS-fetch failures justify the
medium migration.
